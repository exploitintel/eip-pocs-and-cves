## Architecture

```
CVE-YYYY-XXXXX/
├── README.md                    # Full writeup — root cause, attack chain, fix analysis
├── Dockerfile.vulnerable        # Vulnerable target container
├── docker-compose.yml           # One command to stand up the lab
├── intel_brief.md               # CVE intelligence from EIP
├── vulnerability_analysis.md    # Root cause trace and fix assessment
├── lab_build_report.md          # Lab setup details and runtime notes
├── poc_verification_report.md   # Test results: vulnerable vs patched
├── exploit.py / poc.py / *.rb   # Standalone PoC files (layout varies by target)
├── docker-compose-patched.yml   # Optional patched-target lab for bypass analysis
├── bypass_analysis.md           # Optional, when the fix didn't hold
└── screenshots / setup helpers  # Optional assets and helper scripts
```

Each CVE directory is completely self-contained — no shared libraries or cross-CVE dependencies.

## Usage

Every CVE follows the same pattern:

```bash
cd CVE-YYYY-XXXXX

# Build and run the lab
docker compose build
docker compose up -d

# Read the target-specific instructions
sed -n '1,200p' README.md

# Run the exploit command documented for that lab
# Examples vary by target: python3 poc.py, python3 exploit.py, ruby exploit.rb, etc.

# Tear it down
docker compose down
```

Check the individual README for exploit filename, ports, credentials, patched-target workflow, and any required helper setup.

## Deploy

These labs are generated by [CVEForge](https://exploit-intel.com/blog/posts/cveforge-from-shannon-to-autonomous-poc/) — a fork of [Shannon](https://github.com/KeygraphHQ/shannon) rewired to take a CVE number and pointed at the [EIP MCP server](https://exploit-intel.com/blog/posts/introducing-eip-mcp-server/) for vulnerability intelligence.

CVEForge now runs a sequential pipeline with preflight validation, core agent phases, and optional bypass work:

1. **Preflight** — Validate config, workspace, Docker, and credentials
2. **Intel** — Pull CVE intelligence from EIP, clone the repo, identify the vulnerable version
3. **Analysis** — Read the fix diff, trace the vulnerable code path, assess fix completeness
4. **Lab Build** — Generate Dockerfiles and Compose files, build containers, verify they start
5. **PoC & Verify** — Write the exploit and validate it against the vulnerable target
6. **Bypass** *(conditional)* — If the analysis flagged the fix as narrow, try to break the patched target
7. **Report + QA** — Generate the writeup and perform a final quality review

The lab-build phase is usually the bottleneck.

To publish a new CVE entry:
1. Copy the generated `CVE-YYYY-XXXXX/` directory into the repo root
2. Add an entry to the CVE table in `README.md` (sorted by CVSS descending within each score tier)
3. Update the summary line at the bottom of the table if bypass count changed
4. Branch, PR, review — never push directly to `main`

**Never push directly to `main`.** All changes go through a branch and PR.

```bash
git checkout -b <branch-name>
# make changes
git push origin <branch-name>
gh pr create
```

PRs require review before merge.
